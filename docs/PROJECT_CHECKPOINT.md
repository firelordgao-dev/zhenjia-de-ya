# 真的假的鸭项目 Checkpoint

日期：2026-05-25

更新：2026-05-30 已补入诈骗类型识别和按类型固定止损 SOP。

## 当前定位

“真的假的鸭”是一个面向普通群众，尤其是老年人的微信小程序反诈助手。

核心目标：让用户在转账、给验证码、点链接、下载 App、屏幕共享之前，可以快速问一句“真的假的”，并得到适老化、可执行的风险提醒。

## 当前产品形态

主线是微信小程序，不是网页。

- `miniprogram/`：微信小程序前端
- `cloudfunctions/`：微信云函数
- `public/` + `server/`：本地网页原型和视觉预览，仅用于调试和演示

当前本地预览页：

```text
http://localhost:5177/miniprogram-preview.html
```

## 当前视觉方向

品牌主角：无帽呆毛小鸭。

核心符号：

- 放大镜：识别真假
- 蓝盾白勾：安全守护
- 红金绶带、同心书页胸章：公益文化服务气质
- 页面红金细节、书页、同心线：弱化但可感知的民进相关公益文化元素

重要边界：

- 不直接使用民进官方会徽或官方标志
- 不写“中国民主促进会”或“民进”作为 logo 主元素
- 不做政府 App、公安 App 或官方执法工具的视觉感
- 页面可以使用“社会服务 · 反诈守护”这类弱关联表达

## 当前技术链路

### 小程序端

页面：`miniprogram/pages/index/`

能力：

- 第一屏快速入口：粘贴文字、上传截图、紧急止损
- 粘贴微信复制内容
- 上传截图
- OCR 识别截图文字
- 勾选紧急状态
- 勾选紧急状态后即时展示止损卡片，不等待 AI 分析
- 调用反诈分析云函数
- 拨打 `110` / `96110`
- 复制给家人
- 转发小程序
- 结果页默认展示短版行动建议，判断依据、危险点和证据清单折叠到详细说明中

### 云函数

`cloudfunctions/analyzeCase`

- 本地规则引擎保底判断
- DeepSeek V4 Flash 基础分析
- 高风险/复杂情况用 DeepSeek V4 Pro 复核
- 识别常见诈骗类型：冒充公检法、客服退款/理赔、刷单返利、投资理财、屏幕共享/远程控制、钓鱼链接
- 判断当前阶段：早期核实、危险操作请求、隐私可能泄露、资金已风险、正在远程控制
- 按诈骗类型输出固定止损 SOP
- 输出个人隐私安全提醒：验证码、银行卡、身份证、人脸识别、密码、屏幕共享、通讯录和相册权限
- 输出风险等级、疑似诈骗类型、当前阶段、判断依据、危险点、官方核实步骤、隐私保护步骤、下一步行动、报警提醒、证据清单、家属文案

`cloudfunctions/ocrImage`

- 小程序上传截图到微信云存储
- 云函数调用微信服务市场 OCR
- 识别完成或失败后尝试删除临时上传截图

## 环境变量

`analyzeCase` 云函数：

- `DEEPSEEK_API_KEY`
- `DEEPSEEK_API_BASE`
- `DEEPSEEK_FLASH_MODEL`
- `DEEPSEEK_PRO_MODEL`

`ocrImage` 云函数：

- `OCR_SERVICE_ID`
- `OCR_SERVICE_API`
- `OCR_TYPE`
- `OCR_DATA_TYPE`

注意：`DEEPSEEK_API_KEY` 只能放在云函数环境变量或本地 `.env`，不要放进小程序前端。

## 明天重点深挖

建议从以下问题展开：

1. 产品逻辑
   - 老人第一次打开时到底应该先做什么？
   - “上传截图”和“粘贴文字”哪个应该作为主入口？
   - 紧急场景是否要先于 AI 分析直接进入止损指引？
   - 家属协助模式是否进入第一版？

2. 反诈 Agent
   - 风险等级怎么定义才稳？
   - 规则引擎和模型判断发生冲突时谁优先？
   - 哪些内容必须强制提醒 `110`？
   - 当前诈骗类型和固定 SOP 是否覆盖第一版高频场景？
   - 隐私风险是否需要单独进入“账户安全检查清单”？

3. OCR 方案
   - 微信服务市场 OCR 的成本、稳定性、准确率
   - 截图是否应默认上传后删除
   - 是否需要支持多图识别

4. 隐私和合规
   - 是否保存分析记录
   - 是否需要用户授权说明页
   - 家属转发是否会泄露隐私
   - 是否需要敏感信息脱敏

5. 微信小程序落地
   - 开通云开发
   - 配置云函数环境变量
   - 部署 `analyzeCase` 和 `ocrImage`
   - 在微信开发者工具里真机预览
